## Overview

Add a persistent memory system to pearls, stored as first-class entities alongside todos in the same file format.

## Core Design

- **Format:** Same file-based format as todos (JSON front matter), distinguished by a `type: Memory` field. Todos implicitly have `type: ToDo` (not stored in front matter). Memories explicitly set `type: Memory`.
- **Lifecycle:** Memories can be open or closed. Closed = stale/irrelevant to the codebase.
- **First-class:** Memories have IDs, tags, priorities, claims — just like todos. Titles serve as concise pointers to jog the agent's memory; the body holds the actual substance.
- **Cross-referencing:** A pearl can reference a memory and vice versa.

## Creation

- Triggered by phrases like "Remember this" or "Please remember …" — routed through the Pi extension if running in `pi`, or the CLI otherwise.
- Also available via the pearls skill.

## Indexing

- On agent startup or memory compaction, scan all memory files and build an **index of title + ID only** (no bodies) to keep context lean.
- The agent loads this index into its context so it knows what memories exist and can retrieve a body by ID when needed.

## Done Criteria

- CLI commands to create, list, and manage memories
- Pi extension commands to create a memory
- Skill support ("Please remember …")
- Memory index generation on startup/compaction

## Tooling

- **List memories:** A dedicated tool/command to list memories (index of title + ID), separate from listing todos.
- **View memories:** Reuse the existing `pearls get` tool — no new command needed since memories use the same file format.

## Separation from Todos

- `pearls list` should **not** include memories. They are separate concerns and mixing them would clutter the todo view.
- Memories have their own dedicated list command (`pearls memories` or similar).

## TUI Command: /pearls rem

- **Command:** `/pearls rem` (no arguments)
- **Behavior:** Opens an inline input field in the TUI (similar to the search input in `/pearls`) where the user types the memory title/text
- On submit, creates a memory file with `type: Memory` in front matter
- Shows a notification confirming creation (e.g. "Created memory TODO-xxxx")
- If running without a TUI (`!ctx.hasUI`), require the text as a CLI argument instead
- Agent can also trigger via skill ("Please remember …"), which bypasses the TUI and calls the create logic directly

## Pearls TUI: Memory Filter Toggle

- The `/pearls` TUI selector should have a filter toggle to switch between showing todos and memories
- Default view shows todos (current behavior)
- Toggling shows memories with the same search/select UX
- This replaces the need for a separate `/memories` command — everything lives under `/pearls`

## Refinement Notes

1. **Title/body split:** When creating a memory (e.g. "Please remember …"), put a short summary in the title and the full text in the body.
2. **Filter toggle keybinding:** `Ctrl+Shift+M` to toggle between todos and memories in the `/pearls` TUI. Fits existing `Ctrl+Shift+` pattern (R=refine, W=work).
3. **`pearls memories` CLI output:** Same format as `pearls list`, just filtered to type=Memory entries.
4. **Creation via existing `create`:** No new tool actions needed. Use `pearls create --type memory ...` on the CLI, and the existing `create` action in the tool/TUI with `type: Memory` as a parameter.
5. **Same directory:** Memories live in `.pi/todos/` alongside todos, sharing the ID namespace. Distinguished by the `type` field in front matter.
